이번 구현의 핵심은 RTSP로 들어오는 CCTV 영상을 HLS 형태로 변환하고, 브라우저에서 이를 안정적으로 재생할 수 있도록 구성하는 것이었다.
이제부터는 HLS를 선택한 이유가 아니라, 실제로 어떤 구조로 구현했는지를 중심으로 정리해보려고 한다.

## RSTP to HLS 변환 구현
전체 흐름은 비교적 단순했다.
RTSP로 들어오는 원본 스트림을 받아 HLS 세그먼트와 m3u8 플레이리스트로 변환하고, 브라우저에서는 이 m3u8 파일을 기준으로 영상을 재생하는 구조로 구성했다. 

## 전체 구성

1. RTSP 스트림을 HLS로 변환
2. 플레이리스트 생성 대기
3. `m3u8` 및 세그먼트 파일 저장
4. 세그먼트 옵션 조정

### 1. RTSP 스트림을 HLS로 변환

가장 중요한 부분은 `ffmpeg`를 이용해 RTSP 스트림을 HLS 포맷으로 변환하는 과정이었다.
서버는 카메라로부터 들어오는 RTSP 주소를 입력으로 받아, 이를 `m3u8` 플레이리스트와 `ts` 세그먼트 파일로 생성하도록 구성했다.

```python
def _start_stream(self, camera_id, rtsp_url):
    output_dir = os.path.join(settings.MEDIA_ROOT, "cctv", "live", str(camera_id))
    os.makedirs(output_dir, exist_ok=True)

    playlist_path = os.path.join(output_dir, "index.m3u8")

    cmd = [
        "ffmpeg",
        "-rtsp_transport", "tcp",
        "-i", rtsp_url,
        "-an",
        "-c:v", "copy",
        "-f", "hls",
        "-hls_time", "2",
        "-hls_list_size", "24",
        "-hls_delete_threshold", "12",
        "-hls_flags", "delete_segments+omit_endlist",
        "-hls_segment_filename", os.path.join(output_dir, "seg_%03d.ts"),
        playlist_path,
    ]

    subprocess.Popen(
        cmd,
        stdout=subprocess.DEVNULL,
        stderr=subprocess.DEVNULL,
        start_new_session=True,
    )

    return playlist_path
```
여기서 핵심은 RTSP 영상을 다시 인코딩하지 않고 `-c:v copy` 옵션으로 그대로 복사한 뒤, HLS 형식으로만 변환했다는 점이다.(재인코딩 할 시 무자비한 자원의 소모를 볼 수 있다..)

이렇게 하면 인코딩 비용을 줄일 수 있고, 불필요한 지연도 어느 정도 줄일 수 있다.

또한  `-hls_time`, `-hls_list_size`, `-hls_delete_threshold` 같은 옵션을 통해 세그먼트 길이와 유지 개수를 조정했다.
이 값들은 재생 안정성과 지연 시간에 직접적인 영향을 주기 때문에, HLS 구현에서 꽤 중요한 설정이었다.

### 2. 플레이리스트 생성 대기

`ffmpeg` 프로세스를 실행했다고 해서 즉시 재생 가능한 상태가 되는 것은 아니었다.
브라우저가 재생을 시작하려면 최소한 `m3u8` 플레이리스트 파일이 먼저 생성되어 있어야 했기 때문에, 서버에서는 해당 파일이 준비될 때까지 짧게 대기하는 로직을 두었다.

```python
def wait_for_playlist(self, camera_id, timeout=5, quality=None):
    paths = self._paths(camera_id, quality)
    deadline = time.time() + timeout

    while time.time() < deadline:
        if os.path.exists(paths["playlist"]):
            return True
        time.sleep(0.2)

    return False
```

플레이리스트가 아직 생성되지 않은 시점에 브라우저가 먼저 접근하면 404가 발생하거나 재생 시작이 불안정해질 수 있기 때문이다.
그래서 스트림 시작 직후에는 `m3u8` 파일이 준비되었는지 확인하는 단계가 필요했다.

### 3. 세그먼트 파일 저장

HLS로 변환이 시작되면 결과물은 하나의 영상 파일이 아니라, `index.m3u8` 플레이리스트와 여러 개의 `ts` 세그먼트 파일로 나뉘어 저장된다.
내 구현에서는 카메라별 디렉토리를 따로 두고, 그 안에 플레이리스트와 세그먼트 파일이 함께 생성되도록 구성했다.

예를 들어 디렉토리 구조는 다음과 같은 형태가 된다.
```debug
media/
  cctv/
    live/
      1/
        index.m3u8
        seg_000.ts
        seg_001.ts
        seg_002.ts
        ...
```
### 4. 세그먼트 옵션 조정

HLS 변환 자체는 `ffmpeg` 명령으로 시작되지만, 실제로는 세그먼트 관련 옵션을 어떻게 조정하느냐가 더 중요했다.

`-hls_time 2`는 세그먼트 길이를 2초로 설정하는 옵션이다.
세그먼트 길이가 너무 길면 지연 시간이 커지고, 반대로 너무 짧으면 파일이 지나치게 자주 생성되어 관리 비용이 증가할 수 있다.

`-hls_list_size 24`는 플레이리스트에 유지할 세그먼트 개수를 의미한다.
값이 너무 작으면 재생 구간이 짧아져 불안정해질 수 있고, 너무 크면 지연과 저장 비용이 함께 늘어날 수 있다.

`-hls_delete_threshold 12`와 `delete_segments` 옵션은 오래된 세그먼트를 정리하기 위한 설정이다.
라이브 스트리밍 특성상 세그먼트 파일은 계속 누적되기 때문에, 이 옵션이 없으면 디스크 사용량이 계속 증가하게 된다.

정리하면, RTSP를 HLS로 변환하는 구현 자체는 비교적 단순했지만, 실제 서비스에서 안정적으로 사용하려면 세그먼트 길이, 유지 개수, 삭제 시점 같은 옵션을 함께 조정하는 과정이 중요했다.
