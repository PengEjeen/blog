# 7. 운영 적용 체크리스트

로컬에서 Keycloak 로그인이 성공했다고 해서 바로 운영에 올릴 수 있는 것은 아니다. Keycloak은 인증 서버이기 때문에 장애가 나면 전체 서비스 로그인이 막히고, 설정이 잘못되면 보안 사고로 이어질 수 있다.

운영 적용 전에는 아래 항목을 최소 기준으로 점검한다.

## 7.1 HTTPS 강제
운영 환경에서는 `http`를 사용하면 안 된다. 세션 쿠키가 탈취되면 실제 사용자로 요청을 보낼 수 있기 때문에 반드시 HTTPS 위에서 주고받아야 한다.

Keycloak을 reverse proxy 뒤에 둘 경우 외부 요청은 HTTPS인데 Keycloak 내부에서는 HTTP로 보일 수 있다. 이때 proxy header 설정이 틀리면 issuer URL이나 redirect URL이 잘못 생성된다.

확인할 항목:
- 외부 접속 URL이 HTTPS인지
- Keycloak hostname 설정과 issuer가 일치하는지
- reverse proxy에서 `X-Forwarded-*` 헤더가 올바르게 전달되는지

## 7.2 세션 만료 정책
세션 방식에서는 Django 세션 만료와 Keycloak SSO 세션 만료를 함께 봐야 한다. 둘 중 하나만 길게 잡으면 사용자가 예상하지 못한 로그인 유지 또는 재로그인 상황을 만날 수 있다.

예를 들어 Django 세션은 살아 있는데 Keycloak SSO 세션이 끝나 있으면, 앱 안에서는 로그인된 것처럼 보이지만 SSO 기반 재인증이 필요한 순간 문제가 생길 수 있다. 반대로 Django 세션은 만료됐는데 Keycloak SSO 세션이 살아 있으면 로그인 버튼을 눌렀을 때 비밀번호 없이 다시 들어온다.

운영에서 볼 기준:
- Django `SESSION_COOKIE_AGE`
- Keycloak SSO Session Idle
- Keycloak SSO Session Max
- 관리자 세션: 일반 사용자보다 더 엄격하게 설정

## 7.3 관리자 계정 보호
Keycloak 관리자 계정은 전체 인증 체계를 바꿀 수 있는 권한을 가진다. 관리자 계정이 탈취되면 사용자 role, client secret, redirect URI, identity provider 설정이 모두 위험해진다.

최소한 관리자 계정에는 MFA를 적용해야 한다. 또한 운영 Admin Console은 사내망, VPN, 특정 IP에서만 접근하도록 제한하는 것이 좋다.

점검 항목:
- 기본 `admin` 계정 제거 또는 이름 변경
- 관리자 MFA 적용
- 관리자 계정 수 최소화
- Admin Console 접근 IP 제한
- 관리자 작업 로그 확인 체계 마련

## 7.4 Realm 백업
Keycloak 설정은 DB에 저장된다. DB 백업도 중요하지만, realm export를 별도로 남겨두면 설정 변경 이력을 추적하고 장애 시 복구하기 쉽다.

백업 대상:
- Realm 설정
- Client 설정
- Role, Group 설정
- Client Scope와 Mapper
- Identity Provider 설정

사용자 데이터까지 export할지는 상황에 따라 다르다. 개인정보와 보안 정책을 고려해야 하므로 운영에서는 realm 설정 백업과 DB 백업 정책을 분리해서 설계하는 것이 좋다.

## 7.5 Client Secret 관리
Confidential client를 사용하면 client secret이 발급된다. 이 값은 비밀번호와 동일하게 취급해야 한다. Git에 올리거나 프런트엔드 코드에 포함하면 안 된다.

운영에서는 secret을 환경 변수, secret manager, Kubernetes Secret 같은 안전한 저장소에서 주입한다. secret rotation 절차도 미리 정해두는 것이 좋다.

## 7.6 쿠키와 CSRF
세션 기반 구조에서는 쿠키 보안 설정이 인증 품질을 좌우한다. 운영에서는 `HttpOnly`, `Secure`, `SameSite`를 명확히 정해야 한다.

기본 점검 항목:
- `SESSION_COOKIE_HTTPONLY = True`
- `SESSION_COOKIE_SECURE = True`
- `CSRF_COOKIE_SECURE = True`
- 상태 변경 API에 CSRF 보호 적용
- 프런트와 백엔드가 다른 도메인일 때 CORS와 cookie domain 확인

`SameSite=Lax`는 일반적인 웹 로그인 흐름에서 다루기 쉽다. 프런트와 백엔드가 완전히 다른 사이트로 분리되어 쿠키를 cross-site로 보내야 한다면 `SameSite=None; Secure`가 필요하고, 그만큼 CSRF 방어를 더 신중하게 설계해야 한다.

## 7.7 모니터링
인증 서버는 장애가 나면 체감 영향이 크다. Keycloak 자체의 상태뿐 아니라 DB 연결, 로그인 실패율, callback 실패율, 관리자 로그인 이벤트를 관찰해야 한다.

우선순위 높은 지표:
- Keycloak process health
- PostgreSQL 연결 상태
- 로그인 성공/실패 이벤트
- 4xx/5xx 응답 비율
- callback 실패율
- Keycloak endpoint latency
- 디스크 사용량과 DB 백업 성공 여부

운영에서 Keycloak은 "한 번 띄워두면 끝나는 도구"가 아니다. 사용자 인증의 관문이므로, 애플리케이션 API와 같은 수준으로 배포, 백업, 모니터링을 가져가야 한다.
