# 9. 정리

Keycloak 도입의 핵심은 "로그인 기능 추가"가 아니라 "인증/인가 정책을 애플리케이션 코드 밖으로 분리"하는 데 있다.

내 경험 기준으로는 아래 3가지만 지키면 시행착오가 크게 줄었다.

1. 프런트는 `Authorization Code + PKCE` 고정
2. 백엔드는 Keycloak callback을 처리한 뒤 Django 세션 생성
3. redirect/logout URL과 세션 쿠키 정책을 초기에 표준화

이 세 가지를 지키면 구조가 단순해진다. 프런트엔드는 로그인 상태를 `/auth/me`로 확인하고, 백엔드는 Django 세션을 기준으로 요청을 처리한다. 사용자 인증, SSO 정책, role 관리는 Keycloak에서 중앙 관리한다.

Keycloak은 강력하지만 설정 중심 도구라서 대충 붙이면 나중에 문제가 커진다. 특히 redirect URI, logout redirect URI, client secret, session cookie 설정은 초기에 정확히 맞춰야 한다. 이 값들은 프런트엔드, 백엔드, Keycloak 설정에 모두 걸쳐 있기 때문에 한 곳만 틀려도 인증 흐름이 깨진다.

## 9.1 도입 순서
처음 도입할 때는 아래 순서로 진행하는 것이 가장 덜 헷갈렸다.

1. 로컬에서 Keycloak과 PostgreSQL을 실행한다.
2. 애플리케이션용 realm을 만든다.
3. 백엔드용 confidential client를 만든다.
4. callback URI와 logout redirect URI를 등록한다.
5. 테스트 사용자를 만들고 role을 부여한다.
6. React에서 `/auth/login`, `/auth/me`, `/auth/logout` 흐름을 붙인다.
7. Django에서 callback을 처리하고 세션을 만든다.
8. 세션에 저장한 role 또는 내부 사용자 권한으로 접근 제어를 붙인다.
9. 운영 전 HTTPS, CSRF, 관리자 MFA, 백업, 모니터링을 점검한다.

## 9.2 직접 구현과 Keycloak의 차이
직접 세션 로그인을 구현하는 방식은 작은 프로젝트에서는 빠르다. 하지만 인증 요구사항이 늘어나면 로그인 API가 점점 인증 플랫폼처럼 변한다. 소셜 로그인, SSO, role 관리, 세션 정책, 계정 잠금, MFA를 모두 직접 구현하기 시작하면 비즈니스 코드보다 인증 코드가 더 복잡해진다.

Keycloak은 이 복잡도를 애플리케이션 밖으로 빼준다. 대신 Keycloak 자체를 운영해야 하는 비용이 생긴다. 그래서 Keycloak 도입은 "무료로 인증을 해결한다"가 아니라 "인증 책임을 전문 도구로 분리하고, 그 도구를 운영한다"에 가깝다.

## 9.3 다음에 더 볼 주제
기본 연동이 끝났다면 다음 주제까지 이어서 보면 실무 적용에 도움이 된다.

- Keycloak behind reverse proxy 설정
- Terraform으로 realm/client 관리
- Group 기반 권한 모델
- Group 기반 권한과 Django permission 매핑
- SSO 세션과 Django 세션 만료 정책
- Keycloak event listener와 감사 로그
- Kubernetes 환경에서 Keycloak 운영
