# 4. React에서 세션 로그인 붙이기

내가 사용한 방식은 React가 Keycloak 토큰을 직접 들고 다니는 구조가 아니었다. React는 로그인 버튼을 누르면 백엔드의 로그인 endpoint로 이동하고, 백엔드가 Keycloak으로 redirect한다. 로그인 성공 후에는 Django가 서버 세션을 만들고, React는 세션 쿠키로 API를 호출한다.

이 방식의 핵심은 브라우저에 Access Token이나 Refresh Token을 저장하지 않는다는 점이다. 프런트엔드는 인증 상태를 직접 판정하지 않고, 백엔드의 `/auth/me` 같은 endpoint를 통해 현재 로그인 사용자를 확인한다.

## 4.1 전체 흐름

세션 기반 Keycloak 로그인 흐름은 아래처럼 움직인다.

1. 사용자가 React에서 로그인 버튼을 누른다.
2. React가 `GET /auth/login`으로 이동한다.
3. Django가 Keycloak authorization endpoint로 redirect한다.
4. 사용자가 Keycloak에서 로그인한다.
5. Keycloak이 `GET /auth/callback?code=...`로 돌려보낸다.
6. Django가 code를 검증하고 사용자 정보를 확인한다.
7. Django가 서버 세션을 생성하고 세션 쿠키를 내려준다.
8. React는 이후 API 요청에 세션 쿠키를 포함한다.

React 입장에서는 일반적인 세션 기반 로그인과 거의 같다. 차이는 로그인 화면과 사용자 인증을 Django가 직접 처리하지 않고 Keycloak에 위임한다는 점이다.

## 4.2 로그인 버튼

React에서 로그인 버튼은 Keycloak 주소를 직접 알 필요가 없다. 백엔드가 어떤 realm, client, redirect URI를 쓰는지 알고 있기 때문이다.

```tsx
export function LoginButton() {
  const handleLogin = () => {
    window.location.href = "/auth/login";
  };

  return (
    <button type="button" onClick={handleLogin}>
      로그인
    </button>
  );
}
```

프런트가 Keycloak URL을 직접 조립하지 않으면 설정 변경에 강해진다. realm 이름, client id, redirect URI가 바뀌어도 React 코드를 다시 배포할 필요가 적다.

## 4.3 현재 사용자 조회

로그인 여부는 백엔드 세션을 기준으로 확인한다. React는 페이지 진입 시 `/auth/me`를 호출하고, 응답이 200이면 로그인된 사용자로 처리한다.

```tsx
type CurrentUser = {
  id: string;
  email: string;
  name: string;
  roles: string[];
};

export async function fetchCurrentUser(): Promise<CurrentUser | null> {
  const response = await fetch("/auth/me", {
    credentials: "include",
  });

  if (response.status === 401) {
    return null;
  }

  if (!response.ok) {
    throw new Error("Failed to fetch current user");
  }

  return response.json();
}
```

여기서 중요한 값은 `credentials: "include"`다. 프런트와 백엔드가 같은 origin이면 기본 동작으로도 쿠키가 붙을 수 있지만, 도메인이나 포트가 다르면 명시적으로 credentials 설정이 필요하다.

## 4.4 API 호출

세션 기반 구조에서는 API 호출마다 `Authorization: Bearer ...` 헤더를 붙이지 않는다. 브라우저가 세션 쿠키를 보내고, 백엔드는 세션 저장소에서 로그인 사용자를 찾는다.

```tsx
export async function sessionFetch(input: RequestInfo, init: RequestInit = {}) {
  const response = await fetch(input, {
    ...init,
    credentials: "include",
    headers: {
      "Content-Type": "application/json",
      ...init.headers,
    },
  });

  if (response.status === 401) {
    window.location.href = "/auth/login";
  }

  return response;
}
```

이 방식은 프런트 코드가 단순해진다. 토큰 만료 시간, refresh token 갱신, token storage 같은 문제를 React에서 직접 다루지 않아도 된다.

## 4.5 로그아웃

로그아웃은 두 단계를 모두 처리해야 한다.

1. Django 애플리케이션 세션 제거
2. Keycloak SSO 세션 종료

React에서는 백엔드 로그아웃 endpoint만 호출한다.

```tsx
export async function logout() {
  window.location.href = "/auth/logout";
}
```

백엔드는 Django 세션을 지운 뒤 Keycloak logout endpoint로 redirect한다. 그래야 애플리케이션에서는 로그아웃됐지만 Keycloak SSO 세션이 살아 있어서 다시 자동 로그인되는 상황을 피할 수 있다.

## 4.6 이 방식의 장단점

세션 방식의 장점은 브라우저에 민감한 토큰을 직접 저장하지 않는다는 점이다. XSS로 localStorage의 token이 털리는 구조를 피할 수 있고, 백엔드가 로그인 상태를 중앙에서 제어하기 쉽다.

대신 세션 쿠키를 쓰므로 CSRF 방어가 중요하다. SameSite, Secure, HttpOnly 설정을 정확히 잡고, 상태 변경 API에는 CSRF token을 적용해야 한다. 또한 프런트와 백엔드가 다른 도메인에 있을 경우 CORS와 cookie domain 설정을 함께 맞춰야 한다.
