# 6. 자주 겪은 장애 4가지

Keycloak 연동에서 어려운 부분은 코드보다 설정이다. 세션 기반 구조에서도 여러 URL, client 설정, callback 처리, 브라우저 redirect가 맞물려 돌아간다. 그래서 설정 하나가 어긋나면 에러 메시지만 보고 원인을 찾기 어렵다.

이 장에서는 내가 실제로 자주 만난 문제를 기준으로 증상, 원인, 해결 방법을 정리한다.

## 6.1 `invalid redirect_uri`
`invalid redirect_uri`는 Keycloak을 처음 붙일 때 가장 자주 만나는 오류다. 사용자가 로그인 페이지까지는 이동하지만, 로그인 직전 또는 로그인 후 돌아오는 과정에서 redirect URI가 허용되지 않았다는 에러가 나온다.

원인은 대부분 Keycloak Client에 등록된 `Valid Redirect URIs`와 실제 브라우저 URL이 다르기 때문이다. `localhost`와 `127.0.0.1`을 혼용하거나, 포트가 다르거나, trailing slash가 다르거나, 프런트 라우팅 경로가 빠진 경우가 많다.

해결:
- Client의 `Valid Redirect URIs`에 실제 값 정확히 등록
- 로컬 개발 URL을 하나로 통일
- 개발 환경에서는 `http://localhost:3000/*`처럼 경로를 열어두되 운영에서는 정확한 도메인만 등록

운영 환경에서 redirect URI를 너무 넓게 열면 보안 문제가 된다. 로컬에서는 편의를 위해 wildcard를 쓸 수 있지만, 운영에서는 `https://app.example.com/*`처럼 신뢰할 수 있는 도메인으로 제한해야 한다.

## 6.2 세션에 role이 안 들어옴
로그인은 성공했고 사용자에게 role도 부여했는데, Django 세션이나 사용자 모델에는 권한 정보가 없는 경우가 있다. 이 문제는 권한 체크를 붙이는 시점에 자주 드러난다.

원인은 두 가지로 나뉜다. 첫째, Keycloak이 role 정보를 OIDC 응답에 내려주지 않는 설정일 수 있다. 둘째, Django callback 코드가 userinfo나 claim에서 role 정보를 읽어 세션에 저장하지 않았을 수 있다.

해결:
- Realm roles mapper 확인
- `realm_access`/`resource_access` claim 포함 여부 점검
- Django callback에서 어떤 사용자 정보를 세션에 저장하는지 확인
- 백엔드 권한 체크 기준을 Realm Role로 할지 Group으로 할지 먼저 결정

세션 방식에서는 로그인 시점에 필요한 권한을 세션에 복사해두는 경우가 많다. 다만 권한 변경이 즉시 반영되어야 하는 서비스라면 세션을 짧게 가져가거나, 중요한 API에서 Keycloak 또는 내부 권한 DB를 다시 확인하는 구조가 필요하다.

## 6.3 로그아웃 후 앱으로 복귀 안 됨
로그아웃 버튼을 눌렀을 때 Keycloak 세션은 종료되지만 앱으로 돌아오지 못하는 문제가 있다. 브라우저가 Keycloak 로그아웃 endpoint에 머물거나, redirect 관련 오류가 보인다.

원인은 `post_logout_redirect_uri`가 client 설정에 등록되어 있지 않기 때문이다. 로그인 redirect URI와 로그아웃 redirect URI는 별도로 관리된다.

해결:
- Client의 `Valid Post Logout Redirect URIs` 등록
- 프런트 로그아웃 URL을 등록값과 정확히 일치
- 운영 도메인과 스테이징 도메인을 분리해서 등록

프런트엔드에서 화면 상태만 지우는 방식은 완전한 로그아웃이 아니다. Django 세션과 Keycloak SSO 세션을 모두 종료해야 한다. Keycloak SSO 세션이 살아 있으면 사용자가 다시 로그인 버튼을 눌렀을 때 비밀번호 입력 없이 바로 로그인될 수 있다.

## 6.4 Callback에서 code 교환이 실패함
로그인은 성공했는데 Django callback에서 token endpoint 호출이 실패하는 경우가 있다. 세션 방식에서는 이 단계가 실패하면 Django 세션을 만들 수 없으므로 사용자는 로그인 완료 화면까지 가지 못한다.

주요 원인은 client secret 불일치, redirect URI 불일치, authorization code 재사용, Keycloak과 Django 서버 시간 오차다. authorization code는 한 번만 사용할 수 있으므로 callback endpoint를 새로고침하면 실패할 수 있다.

해결:
- Django의 `KEYCLOAK_CLIENT_SECRET`이 Keycloak client secret과 같은지 확인
- token 요청의 `redirect_uri`가 authorization 요청 때 보낸 값과 같은지 확인
- callback endpoint가 중복 호출되지 않는지 확인
- Keycloak 서버와 Django 서버의 시간을 동기화

callback 실패는 프런트보다 백엔드 로그를 먼저 봐야 한다. Keycloak token endpoint 응답 본문에 `invalid_grant`, `invalid_client`, `redirect_uri` 관련 메시지가 들어 있는 경우가 많다.

## 6.5 장애를 줄이는 설정 습관
Keycloak 장애는 대부분 설정값의 불일치에서 시작한다. 그래서 로컬, 스테이징, 운영의 값을 표로 정리해두는 것만으로도 많은 시간을 줄일 수 있다.

| 항목 | 로컬 | 운영에서 확인할 점 |
| --- | --- | --- |
| Keycloak URL | `http://localhost:8080` | 외부 공개 hostname과 issuer 일치 |
| Redirect URI | `http://localhost:3000/*` | 운영 도메인만 허용 |
| Logout Redirect URI | `http://localhost:3000` | post logout URI 별도 등록 |
| Callback URI | `http://localhost:8000/auth/callback` | client redirect URI와 일치 |
| Session Cookie | Django sessionid | Secure, HttpOnly, SameSite 확인 |

설정 문제를 빨리 찾으려면 브라우저 주소창의 실제 redirect URI, Keycloak client 설정, Django 환경 변수, callback 로그를 함께 비교해야 한다.
