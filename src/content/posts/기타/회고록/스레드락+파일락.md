## 문제 상황

최근 백엔드 작업을 하면서, 하나의 작업이 동시에 여러 번 실행되지 않도록 막아야 하는 상황이 있었다.
겉으로 보기에는 단순히 실행 요청을 한 번만 처리하면 되는 문제처럼 보였지만, 실제로는 같은 요청이 거의 동시에 들어오거나, 서버가 여러 프로세스로 떠 있는 상황까지 고려해야 했다.

처음에는 애플리케이션 내부에서 `threading.Lock()`만 사용해도 충분할 것 같았다.
같은 프로세스 안에서 여러 스레드가 동시에 같은 로직에 접근하는 것만 막을 수 있다면 문제를 해결할 수 있다고 생각했기 때문이다.

하지만 실제 운영 환경에서는 그 정도로는 충분하지 않았다.
개발 서버나 단일 프로세스 환경에서는 문제가 잘 드러나지 않지만, 멀티 프로세스 환경에서는 서로 다른 프로세스가 같은 작업을 동시에 실행할 수 있기 때문이다.
즉, 프로세스 내부의 동시성만 제어해서는 중복 실행 문제를 완전히 막을 수 없었다.

## 처음에는 스레드 락만 사용했다

가장 먼저 적용한 것은 스레드 락이었다.

```python
self._lock = threading.Lock()
```
이 락은 같은 프로세스 안에서 여러 스레드가 동시에 특정 로직에 들어오는 것을 막는 데 유용했다.
예를 들어 거의 동시에 두 요청이 들어오더라도, 한 스레드가 작업을 시작하면 다른 스레드는 락이 해제될 때까지 기다리게 만들 수 있다.

이 방식은 단일 프로세스 기준에서는 꽤 단순하고 효과적이다.
문제는 이 락이 현재 프로세스 안에서만 유효하다는 점이다.

스레드 락만으로는 부족했던 이유

운영 환경에서는 애플리케이션이 하나의 프로세스로만 실행되지 않을 수 있다.
워커가 여러 개 떠 있거나, 서버 재시작 과정에서 이전 프로세스와 새로운 프로세스가 잠깐 동시에 존재할 수도 있다.

이 경우 각 프로세스는 서로 다른 메모리 공간을 사용하기 때문에, 한쪽 프로세스에서 threading.Lock()을 획득했다고 해도 다른 프로세스는 그 사실을 전혀 알지 못한다.
결국 프로세스가 다르면 같은 작업이 다시 실행될 수 있고, 내가 막고 싶었던 '중복 실행' 문제는 여전히 남아 있게 된다.

그래서 파일 락을 함께 사용했다

이 문제를 보완하기 위해 프로세스 간에도 공유할 수 있는 잠금 방식이 필요했고, 여기서 파일 락을 사용했다.
```python
session["lock_file"] = open(paths["lock_path"], "a+b")
return _try_lock_file_nonblock(session["lock_file"])
```
파일 락은 특정 파일을 기준으로 잠금을 획득하기 때문에, 같은 머신에서 실행 중인 다른 프로세스도 동일한 락 상태를 확인할 수 있다.
즉, 어떤 프로세스가 먼저 락을 잡고 작업을 시작했다면, 다른 프로세스는 같은 작업을 다시 시작하지 못하게 막을 수 있다.

내 구현에서는 프로세스 내부에서는 threading.Lock()으로 동시 접근을 제어하고, 프로세스 간에는 lock 파일을 통해 중복 실행을 방지하는 구조를 사용했다.

최종적으로 가져간 구조

정리하면 역할은 다음처럼 나뉘었다.

threading.Lock()은 같은 프로세스 내부에서의 동시 접근을 막는다.

파일 락은 서로 다른 프로세스 사이에서 동일 작업의 중복 실행을 막는다.

즉, 하나의 락으로 모든 문제를 해결하려고 하기보다는, 동시성 문제가 발생하는 범위를 나눠서 각각에 맞는 잠금 방식을 적용한 셈이다.

정리

처음에는 단순히 락 하나만 있으면 충분할 거라고 생각했지만, 실제 운영 환경에서는 스레드와 프로세스를 구분해서 봐야 했다.
스레드 락은 프로세스 내부의 경쟁 상태를 막는 데 적합했고, 파일 락은 프로세스 바깥까지 범위를 넓혀 중복 실행을 제어하는 데 유효했다.

실무에서 동시성 문제를 다룰 때는 '락을 썼다'보다도, 그 락이 어느 범위까지 효력을 가지는지를 먼저 구분하는 것이 더 중요하다는 걸 다시 느꼈다.
