---
title: 16장 TypeError - cannot pickle 'BufferedRandom' instances
date: 2026-04-24
---

# 1. 문제 코드

```python
data = request.data.copy()
```

- `request.data`는 DRF가 제공하는 QueryDict 비슷한 객체.
- 일반 폼 데이터만 있으면 문제 없음.
- **`multipart/form-data`로 파일 업로드가 들어오면** `request.FILES`에 `InMemoryUploadedFile` / `TemporaryUploadedFile` 객체가 포함된다.
- 이 객체 내부에는 파일 핸들 (`BufferedRandom`)이 들어 있다.

# 2. `.copy()` 동작 방식

- Django의 `QueryDict.copy()`는 내부적으로 `deepcopy`를 호출한다.

# 3. `deepcopy` → `pickle` 시도

- `deepcopy`는 객체를 완전히 복사하기 위해 pickle 프로토콜과 비슷한 방식을 사용.
- 이 과정에서 `InMemoryUploadedFile` 내부의 파일 핸들 (`BufferedRandom`)을 직렬화하려고 한다.
- 그런데 `BufferedRandom`은 **운영체제 레벨의 파일 디스크립터**를 잡고 있는 객체라 pickle이 불가능 → `TypeError: cannot pickle 'BufferedRandom' instances` 발생.

# 4. 해결 방향

## 4.1 파일은 분리해서 처리

```python
# 파일은 따로 빼두고
files = request.FILES

# 메타데이터만 deepcopy 가능한 dict로 변환
data = {key: request.data[key] for key in request.data if key not in files}
```

## 4.2 또는 얕은 복사로 충분한지 검토

```python
data = dict(request.data.lists())  # QueryDict → dict, 파일 객체는 참조만 복사
```

## 4.3 Serializer에 그대로 위임

가능하면 `serializer = MySerializer(data=request.data, files=request.FILES)`처럼 DRF 직렬화기에 맡겨 직접 `.copy()`하지 않는다.

# 5. 핵심 교훈

`deepcopy`는 OS 리소스 (파일 디스크립터, 소켓, lock 등)를 잡고 있는 객체에 대해 안전하지 않다. 업로드 파일을 다룰 때는 **메타데이터와 파일 객체를 분리**해서 처리하는 것이 안전하다.
