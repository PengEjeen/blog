# 5. Django에서 Keycloak Session 처리하기

세션 기반 구조에서는 Django가 OIDC client 역할을 한다. React가 Keycloak token을 직접 받는 것이 아니라, Django가 Keycloak callback을 처리하고 Django session을 생성한다.

이 방식에서도 OIDC 내부적으로는 authorization code와 token 교환이 일어난다. 하지만 이 token은 백엔드 내부에서 사용자 확인 용도로만 사용하고, 브라우저에는 Django 세션 쿠키만 내려준다. 그래서 프런트엔드는 Keycloak token을 저장하거나 갱신하지 않는다.

## 5.1 필요한 설정

Django 쪽에는 Keycloak realm, client id, client secret, redirect URI가 필요하다.

```python
# config/settings.py
KEYCLOAK_BASE_URL = "http://localhost:8080"
KEYCLOAK_REALM = "myapp"
KEYCLOAK_CLIENT_ID = "myapp-web"
KEYCLOAK_CLIENT_SECRET = "change-me"
KEYCLOAK_REDIRECT_URI = "http://localhost:8000/auth/callback"

KEYCLOAK_ISSUER = f"{KEYCLOAK_BASE_URL}/realms/{KEYCLOAK_REALM}"
KEYCLOAK_AUTH_URL = f"{KEYCLOAK_ISSUER}/protocol/openid-connect/auth"
KEYCLOAK_TOKEN_URL = f"{KEYCLOAK_ISSUER}/protocol/openid-connect/token"
KEYCLOAK_USERINFO_URL = f"{KEYCLOAK_ISSUER}/protocol/openid-connect/userinfo"
KEYCLOAK_LOGOUT_URL = f"{KEYCLOAK_ISSUER}/protocol/openid-connect/logout"
```

`KEYCLOAK_CLIENT_SECRET`은 절대 Git에 올리면 안 된다. 운영에서는 환경 변수나 secret manager에서 주입해야 한다.

## 5.2 로그인 시작

`/auth/login`은 사용자를 Keycloak authorization endpoint로 redirect한다. 여기서 `state`는 CSRF 방어와 callback 검증을 위해 세션에 저장한다.

```python
# app/views.py
import secrets
from urllib.parse import urlencode

from django.conf import settings
from django.shortcuts import redirect


def login(request):
    state = secrets.token_urlsafe(32)
    request.session["keycloak_state"] = state

    params = {
        "client_id": settings.KEYCLOAK_CLIENT_ID,
        "response_type": "code",
        "scope": "openid email profile",
        "redirect_uri": settings.KEYCLOAK_REDIRECT_URI,
        "state": state,
    }

    return redirect(f"{settings.KEYCLOAK_AUTH_URL}?{urlencode(params)}")
```

이 endpoint는 인증을 직접 하지 않는다. 단지 Keycloak으로 보내는 관문 역할을 한다.

## 5.3 Callback 처리

Keycloak 로그인에 성공하면 Django callback URL로 `code`와 `state`가 넘어온다. Django는 먼저 state가 세션에 저장된 값과 같은지 확인한다.

```python
import requests
from django.http import HttpResponseBadRequest


def callback(request):
    code = request.GET.get("code")
    state = request.GET.get("state")

    if not code or state != request.session.get("keycloak_state"):
        return HttpResponseBadRequest("Invalid login callback")

    token_response = requests.post(
        settings.KEYCLOAK_TOKEN_URL,
        data={
            "grant_type": "authorization_code",
            "client_id": settings.KEYCLOAK_CLIENT_ID,
            "client_secret": settings.KEYCLOAK_CLIENT_SECRET,
            "code": code,
            "redirect_uri": settings.KEYCLOAK_REDIRECT_URI,
        },
        timeout=5,
    )
    token_response.raise_for_status()
    tokens = token_response.json()

    userinfo_response = requests.get(
        settings.KEYCLOAK_USERINFO_URL,
        headers={"Authorization": f"Bearer {tokens['access_token']}"},
        timeout=5,
    )
    userinfo_response.raise_for_status()
    userinfo = userinfo_response.json()

    request.session["user"] = {
        "id": userinfo["sub"],
        "email": userinfo.get("email"),
        "name": userinfo.get("name") or userinfo.get("preferred_username"),
    }
    request.session.pop("keycloak_state", None)

    return redirect("/")
```

여기서 Access Token은 백엔드가 Keycloak userinfo endpoint를 호출하기 위해 잠깐 사용한다. 이 값을 브라우저에 내려주지 않는 것이 세션 기반 구조의 핵심이다.

## 5.4 현재 사용자 API

React는 로그인 상태를 직접 알 수 없으므로 `/auth/me` 같은 API로 현재 세션 사용자를 확인한다.

```python
from django.http import JsonResponse


def me(request):
    user = request.session.get("user")

    if not user:
        return JsonResponse({"detail": "Authentication required"}, status=401)

    return JsonResponse(user)
```

실제 프로젝트에서는 `request.session["user"]`만 쓰기보다 Django `User` 모델과 연결하는 편이 좋다. Keycloak의 `sub`를 외부 식별자로 저장해두면 이메일이 바뀌어도 같은 사용자를 안정적으로 찾을 수 있다.

## 5.5 로그아웃

로그아웃은 Django 세션과 Keycloak SSO 세션을 모두 정리해야 한다.

```python
from urllib.parse import urlencode


def logout(request):
    request.session.flush()

    params = {
        "client_id": settings.KEYCLOAK_CLIENT_ID,
        "post_logout_redirect_uri": "http://localhost:3000",
    }

    return redirect(f"{settings.KEYCLOAK_LOGOUT_URL}?{urlencode(params)}")
```

Keycloak client 설정에는 `post_logout_redirect_uri`가 등록되어 있어야 한다. 이 값이 빠지면 로그아웃 후 앱으로 돌아오지 못하는 문제가 생긴다.

## 5.6 세션 방식에서 중요한 보안 설정

세션 방식은 브라우저에 token을 저장하지 않는 대신 cookie 보안 설정이 중요하다.

```python
# config/settings.py
SESSION_COOKIE_HTTPONLY = True
SESSION_COOKIE_SECURE = True
SESSION_COOKIE_SAMESITE = "Lax"
CSRF_COOKIE_SECURE = True
```

운영에서는 HTTPS가 필수이므로 `SESSION_COOKIE_SECURE`와 `CSRF_COOKIE_SECURE`를 켜야 한다. 프런트와 백엔드가 다른 도메인이라면 `SameSite=None; Secure`가 필요할 수 있고, 이 경우 CSRF 방어를 더 꼼꼼하게 봐야 한다.

세션 기반 구조에서 백엔드는 매 요청마다 JWT를 검증하지 않는다. 대신 Django 세션 저장소를 신뢰한다. 따라서 세션 만료 시간, 세션 저장소 안정성, CSRF 보호, logout 처리가 인증 품질을 좌우한다.
