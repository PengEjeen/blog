# 8. 빠른 검증 커맨드

Keycloak 연동이 막혔을 때는 Admin Console만 보지 말고 OIDC endpoint를 직접 확인하는 것이 빠르다. 세션 기반 구조에서도 Django는 Keycloak authorization endpoint, token endpoint, userinfo endpoint를 사용한다. 먼저 Keycloak이 어떤 메타데이터를 제공하는지 확인해야 한다.

이 장의 명령어는 로컬 검증용이다. 운영 client secret이나 authorization code를 터미널 히스토리에 남기지 않도록 주의해야 한다.

## 8.1 OIDC Discovery 확인

```bash
# OIDC Discovery
curl -s http://localhost:8080/realms/myapp/.well-known/openid-configuration | jq .
```

OIDC Discovery 문서는 해당 realm의 표준 endpoint 목록을 제공한다. 여기서 issuer, authorization endpoint, token endpoint, jwks uri를 확인할 수 있다.

특히 `issuer` 값은 백엔드 설정의 `KEYCLOAK_ISSUER`와 정확히 일치해야 한다. 운영에서 reverse proxy 설정이 잘못되면 issuer가 내부 주소로 나오거나 http로 나오는 경우가 있다.

확인할 값:
- `issuer`
- `authorization_endpoint`
- `token_endpoint`
- `jwks_uri`
- `end_session_endpoint`

## 8.2 Userinfo endpoint 확인

세션 방식에서는 Django callback에서 Keycloak userinfo endpoint를 호출해 사용자 정보를 가져올 수 있다. 실제 호출에는 access token이 필요하지만, 브라우저에 저장하지 않고 백엔드 내부에서만 사용한다.

Discovery 문서의 `userinfo_endpoint` 값을 먼저 확인한다.

```bash
curl -s http://localhost:8080/realms/myapp/.well-known/openid-configuration \
  | jq .userinfo_endpoint
```

## 8.3 JWKS 확인

```bash
# JWKS 확인
curl -s http://localhost:8080/realms/myapp/protocol/openid-connect/certs | jq .
```

JWKS는 OIDC token 서명 검증에 사용할 공개키 목록이다. 세션 방식에서 백엔드가 매 API 요청마다 JWT를 검증하지 않더라도, OIDC 내부 동작을 이해하거나 ID token 검증이 필요한 경우에는 JWKS를 확인하게 된다.

JWKS가 조회되지 않으면 Keycloak URL, realm 이름, 네트워크 접근 권한을 먼저 확인한다.

## 8.4 Token endpoint 확인
Django callback은 authorization code를 token endpoint로 보내 사용자 확인에 필요한 응답을 받는다. 로컬 테스트에서는 password grant를 잠시 사용할 수 있지만, 운영 애플리케이션 로그인 방식으로 권장되지는 않는다.

```bash
curl -s \
  -X POST "http://localhost:8080/realms/myapp/protocol/openid-connect/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=password" \
  -d "client_id=myapp-web" \
  -d "username=test@example.com" \
  -d "password=password123" | jq .
```

이 방식은 client 설정에서 Direct Access Grants가 켜져 있어야 동작한다. 단, 브라우저 기반 로그인에서는 Authorization Code Flow를 사용해야 하므로 password grant를 운영 표준으로 삼으면 안 된다.

## 8.5 세션 쿠키 확인
로그인이 성공했는데 React에서 계속 비로그인 상태로 보인다면 브라우저 개발자 도구에서 세션 쿠키가 내려왔는지 확인한다.

확인할 항목:
- `sessionid` 쿠키가 존재하는지
- `HttpOnly`가 켜져 있는지
- HTTPS 환경에서 `Secure`가 켜져 있는지
- 프런트와 백엔드 도메인이 다를 때 cookie domain이 맞는지
- `SameSite` 정책 때문에 쿠키가 누락되지 않는지

세션 방식에서 백엔드 API가 401을 반환한다면 token payload보다 Django 세션 저장소와 쿠키 전송 여부를 먼저 확인하는 것이 맞다.
