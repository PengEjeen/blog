# 1. Keycloak 개요

## 1.1 Keycloak이란?
Keycloak은 오픈소스 IAM(Identity and Access Management) 솔루션이다.  
애플리케이션의 로그인, SSO 세션, 사용자 정보, 권한(role) 정책을 중앙에서 관리하게 해준다.

핵심은 앱마다 인증 코드를 직접 구현하지 않고, OIDC/OAuth2 표준 기반 인증 서버를 분리해 쓰는 것이다.

조금 더 쉽게 말하면, Keycloak은 서비스 밖에 따로 세우는 인증 전용 서버다. 사용자는 Keycloak에서 로그인하고, 애플리케이션은 그 결과를 받아 자체 세션을 만든다. 로그인 화면, 비밀번호 정책, 소셜 로그인, SSO, 역할 관리 같은 기능을 서비스 코드 안에 계속 밀어 넣지 않아도 된다.

Keycloak을 붙이면 애플리케이션의 책임이 명확해진다.

| 영역 | Keycloak의 책임 | 애플리케이션의 책임 |
| --- | --- | --- |
| 로그인 | 사용자 인증, 세션 관리 | 로그인 버튼 연결 |
| 세션 | Keycloak SSO 세션 관리 | 애플리케이션 세션 쿠키 관리 |
| 권한 | Role, Group, Client Scope 관리 | API 접근 제어 |
| 연동 | OIDC, SAML, LDAP, 소셜 로그인 | 표준 프로토콜 사용 |

이 구조의 장점은 인증 정책을 애플리케이션 배포 없이 바꿀 수 있다는 점이다. 예를 들어 관리자 MFA를 켜거나, 특정 사용자의 role을 바꾸거나, 사내 LDAP 연동을 추가하는 일을 백엔드 코드 수정 없이 처리할 수 있다.

## 1.2 왜 Keycloak을 쓰는가?
프로젝트 초기에 로그인/권한(세션 생성, 만료, role 관리)을 백엔드에서 직접 구현했다가, 서비스가 커지면서 인증 로직이 기능마다 중복되기 시작했다.

실무에서 가장 컸던 문제는 아래 3가지였다.

1. 웹/모바일/API마다 로그인 상태와 세션 정책이 조금씩 달라 일관성이 깨짐
2. 소셜 로그인/사내 SSO 연동 요구가 들어올 때마다 인증 코드가 비대해짐
3. role/권한 변경 시 배포 없이 중앙에서 제어하기 어려움

그래서 인증/인가 서버를 Keycloak으로 분리했고, 앱은 OIDC 표준만 따르게 정리했다.

직접 구현한 인증은 처음에는 빠르다. 회원 테이블을 만들고, 비밀번호를 해싱하고, 로그인 성공 시 서버 세션을 만들면 기본 기능은 금방 된다. 문제는 시간이 지나면서 요구사항이 인증 서버 수준으로 커진다는 데 있다.

예를 들면 아래 요구사항이 들어오는 순간부터 단순 로그인 API는 계속 복잡해진다.

- 관리자 계정만 MFA를 적용해야 한다.
- 사내 계정은 LDAP 또는 SSO로 로그인해야 한다.
- 프런트엔드, 모바일 앱, 백오피스가 같은 계정 체계를 써야 한다.
- 서비스별로 다른 권한을 발급해야 한다.
- SSO 세션 만료와 애플리케이션 세션 만료 정책을 중앙에서 통제해야 한다.

이런 요구사항은 기능 코드와 분리하는 편이 낫다. 인증은 대부분의 서비스에서 공통으로 필요하지만, 잘못 구현했을 때 보안 사고로 이어지는 영역이다. 그래서 직접 구현보다 검증된 IAM 도구를 두고, 서비스는 Keycloak 로그인 결과를 받아 서버 세션을 만드는 방식으로 단순화하는 것이 운영 관점에서 안정적이다.

## 1.3 Keycloak이 적합한 상황
Keycloak은 특히 여러 애플리케이션이 하나의 계정 체계를 공유해야 할 때 효과가 크다. 백오피스, 사용자 웹, 모바일 앱, 내부 API가 모두 같은 사용자와 role을 봐야 한다면 중앙 인증 서버가 있는 편이 훨씬 관리하기 쉽다.

또한 기업 환경처럼 SSO, LDAP, SAML, OIDC 연동이 필요한 경우에도 Keycloak이 잘 맞는다. Keycloak은 외부 Identity Provider와 연결할 수 있고, 애플리케이션은 OIDC callback을 통해 사용자 정보를 받아 세션을 생성할 수 있다. 애플리케이션 입장에서는 외부 로그인 방식이 무엇이든 "로그인 성공 후 세션 생성" 흐름을 거의 동일하게 유지할 수 있다.

반대로 아주 작은 개인 프로젝트나 인증 요구사항이 단순한 CRUD 서비스라면 Keycloak이 과할 수 있다. Keycloak 자체도 운영해야 하는 서버이기 때문에 DB, 백업, 관리자 계정 보안, 버전 업그레이드까지 책임져야 한다. 인증 요구사항이 작고 앞으로도 확장 가능성이 낮다면 프레임워크 기본 인증을 쓰는 편이 더 단순할 수 있다.

## 1.4 이 시리즈에서 다룰 범위
이 글 시리즈는 Keycloak을 이론적으로만 설명하지 않고, 실제 프로젝트에 붙일 때 필요한 순서로 정리한다.

1. Keycloak이 어떤 문제를 해결하는지 이해한다.
2. 로컬에서 PostgreSQL 기반 Keycloak을 실행한다.
3. Realm, Client, Role의 최소 구조를 만든다.
4. React에서 세션 기반 로그인 흐름을 붙인다.
5. Django에서 Keycloak callback을 받아 서버 세션을 만든다.
6. 실무에서 자주 만나는 설정 오류를 정리한다.

목표는 "로그인이 된다"에서 끝나는 것이 아니다. 프런트엔드, 백엔드, Keycloak SSO 세션의 경계를 정확히 나누고, 운영 환경에서도 깨지지 않는 인증 구조를 만드는 것이 목표다.
