# 3. Realm / Client / Role 최소 설정

Keycloak을 처음 만지면 용어가 가장 헷갈린다. 특히 Realm, Client, Role의 경계를 제대로 이해하지 못하면 로그인은 되는데 세션에 권한 정보가 없거나, redirect URL이 계속 실패하거나, callback 처리에서 사용자 정보를 제대로 만들지 못하는 문제가 생긴다.

이 장에서는 실제 프로젝트에서 가장 먼저 잡아야 하는 최소 구조를 정리한다.

## 3.1 Realm
Realm은 Keycloak에서 가장 큰 논리적 격리 단위다. 사용자, client, role, group, 인증 정책이 realm 안에 묶인다. 같은 Keycloak 서버 안에서도 realm이 다르면 사용자와 설정은 서로 분리된다.

보통 `master` realm은 Keycloak 관리자용으로 두고, 애플리케이션용 realm을 새로 만든다. 예를 들어 서비스 이름이 `myapp`이라면 `myapp` realm을 만들고 그 안에 웹 클라이언트, API 클라이언트, 사용자 role을 둔다.

실무에서는 환경별 realm을 나눌 수도 있다.

| 환경 | Realm 예시 | 설명 |
| --- | --- | --- |
| 로컬 | `myapp-local` | 개발자 PC 테스트 |
| 스테이징 | `myapp-stg` | QA, 배포 전 검증 |
| 운영 | `myapp-prod` | 실제 사용자 로그인 |

환경별로 realm을 나누면 redirect URI, token lifespan, SMTP, identity provider 설정을 독립적으로 관리할 수 있다. 다만 설정이 많아질수록 drift가 생기기 쉬우므로 export/import 또는 IaC 방식으로 관리하는 것이 좋다.

## 3.2 Client
Client는 Keycloak 입장에서 "로그인을 요청하는 애플리케이션"이다. 세션 기반 구조에서는 보통 브라우저가 직접 Keycloak client가 되는 것이 아니라, 백엔드가 OIDC client 역할을 맡고 로그인 callback을 처리한다.

내가 보통 처음에 만드는 구조는 아래처럼 단순하게 시작한다.

1. Realm: `myapp`
2. Client(백엔드): `myapp-web` (`confidential`)
3. Redirect URI: `http://localhost:8000/auth/callback`
4. Realm Role: `user`, `admin`
5. 테스트 사용자 1명 생성 후 role 부여

이 구조에서는 React가 Keycloak 토큰을 직접 들고 있지 않는다. React는 `/auth/login` 같은 백엔드 endpoint로 이동시키고, 백엔드가 Keycloak으로 redirect한다. 로그인 성공 후 Keycloak은 백엔드 callback URL로 돌아오고, 백엔드는 사용자 정보를 확인한 뒤 Django 세션을 만든다.

백엔드 서버는 client secret을 안전하게 보관할 수 있으므로 `confidential` client를 사용할 수 있다. 브라우저에 secret이나 access token을 노출하지 않는다는 점이 세션 기반 구조의 가장 큰 장점이다.

## 3.3 Role
Role은 사용자가 시스템에서 어떤 권한을 갖는지 표현하는 값이다. Keycloak에는 크게 Realm Role과 Client Role이 있다.

Realm Role은 realm 전체에서 공통으로 쓰는 권한이다. `user`, `admin`처럼 여러 client에서 공유하는 권한은 Realm Role로 두면 단순하다.

Client Role은 특정 client에 종속된 권한이다. 예를 들어 `myapp-api`에만 필요한 `report:read`, `report:write` 같은 세부 권한은 Client Role로 관리할 수 있다.

처음에는 Realm Role로 시작하는 편이 이해하기 쉽다. 권한 모델이 커지면 Client Role이나 Group 기반 권한으로 확장하면 된다.

## 3.4 Client Scope와 Mapper
로그인에 성공했는데 Django 세션에 role 정보가 없을 수 있다. 이때 대부분은 role 자체가 없는 것이 아니라, 백엔드가 callback 처리 시 role claim을 읽지 않았거나, Keycloak에서 role claim을 내려주지 않는 설정인 경우가 많다.

Client Scope는 어떤 사용자 정보를 OIDC 응답에 포함할지 결정하는 설정이다. Mapper는 Keycloak 내부의 사용자 속성, role, group 같은 정보를 claim으로 변환하는 규칙이다.

세션 기반 구조에서도 백엔드는 로그인 callback 시점에 role claim을 읽어 Django 세션이나 사용자 모델에 반영할 수 있다. 권한을 매 요청마다 Keycloak에 물어볼 수도 있지만, 일반적인 웹 서비스에서는 로그인 시점에 필요한 권한을 세션에 담고, 중요한 권한 변경은 세션 만료나 강제 로그아웃 정책으로 제어한다.

## 3.5 실무 팁
- 브라우저에는 Keycloak access token을 저장하지 않음
- 백엔드는 `confidential client + Authorization Code Flow`
- Redirect URI는 개발/스테이징/운영 URL을 정확히 분리해서 등록
- role 이름은 코드에서 직접 흩뿌리지 말고 상수 또는 정책 레이어에서 관리
- 운영 realm 설정은 수동 클릭만 믿지 말고 export 파일로 백업

Keycloak 설정은 처음에는 UI에서 클릭하며 이해하는 것이 빠르다. 하지만 운영에 들어가면 "누가 언제 어떤 client 설정을 바꿨는지"가 중요해진다. 그래서 최소한 realm export를 주기적으로 남겨야 하고, 가능하다면 Terraform provider나 Keycloak Admin API로 설정을 코드화하는 편이 좋다.
